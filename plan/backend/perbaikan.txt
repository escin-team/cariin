C1 — Desain alur topup fundamental salah
Endpoint ini punya authMiddleware (user session) tapi plan bilang amount sudah "diverifikasi dari payment gateway" — ini dua hal yang bertentangan
wallet.service.tswallet.router.ts
▲
Masalah: Phase 1 blueprint menyatakan dengan jelas: "Sistem Cariin mencatat top-up berdasarkan konfirmasi webhook dari Gateway". Artinya balance wallet baru dikreditkan setelah webhook dari bank/payment gateway masuk — BUKAN dari request langsung user. Plan saat ini mencampur dua alur berbeda menjadi satu endpoint:
Jika ini endpoint user-facing (user klik "Top Up") → user seharusnya TIDAK bisa langsung mengkreditkan balance. Ini harus membuat record PENDING dan mengembalikan VA/QRIS untuk dibayar.
Jika ini endpoint internal dari webhook handler → tidak boleh pakai authMiddleware (user cookie auth). Webhook handler memanggil ini secara programatik, tidak ada user session.
Solusi yang benar — dua endpoint terpisah:
// Endpoint 1: User-facing — INITIATE topup
POST /v1/wallet/topup/initiate
  → authMiddleware (user session)
  → Buat WalletTransaction { status: 'PENDING', amount, method }
  → Return: { vaNumber, qrisUrl, expiresAt }
  → Balance BELUM berubah

// Endpoint 2: Internal — CONFIRM topup (dipanggil webhook handler)
POST /v1/wallet/topup/confirm  (atau dari dalam webhook handler)
  → Tidak pakai authMiddleware user — pakai internal secret
  → Verifikasi referenceId dari payment gateway
  → Idempotency check
  → Credit balance (UPDATE wallet SET balance = balance + amount)
  → Update WalletTransaction { status: 'COMPLETED' }

H2 — BigInt coercion dari JSON tidak dispecifikasikan di schema
JSON tidak support BigInt — "BigInt-compatible number" ambigu tanpa strategi konversi yang jelas
wallet.schema.tswallet.service.ts
▲
JSON native tidak bisa represent BigInt. {"amount": 10000} akan di-parse sebagai JS number. Plan bilang "BigInt-compatible number" tapi tidak menjelaskan di mana dan bagaimana konversi terjadi. Strategi yang benar:
// wallet.schema.ts — terima sebagai number di JSON, validasi range
export const TopupWalletSchema = z.object({
  amount: z.number()
    .int('Amount harus bilangan bulat')
    .min(10_000, 'Minimum topup Rp 10.000')
    .max(10_000_000, 'Maksimum topup Rp 10.000.000'),
  referenceId: z.string().optional(),
});

// wallet.service.ts — konversi ke BigInt di boundary service
export type TopupWalletDto = z.infer<typeof TopupWalletSchema>;

async topup(body: TopupWalletDto, userId: string, idempotencyKey: string) {
  const amountBigInt = BigInt(body.amount); // ← konversi di sini, bukan di schema
  // ... gunakan amountBigInt untuk semua DB operations
}
Ini harus eksplisit di plan agar implementer tidak bingung di mana konversi terjadi.
H3 — balanceBefore / balanceAfter timing tidak dispecifikasikan
Schema punya kedua field ini tapi service flow tidak menjelaskan kapan nilainya diambil
wallet.service.ts
▲
Model WalletTransaction punya balanceBefore dan balanceAfter tapi service flow hanya menyebutkan "CREATE WalletTransaction record" tanpa menjelaskan bagaimana nilai-nilai ini dihitung. Nilai ini harus diambil dari hasil SELECT FOR UPDATE (step 4) di plan:
// Step 4: Pessimistic lock — simpan balance saat ini
const wallet = await tx.$queryRaw<[{balance: bigint}]>`
  SELECT balance FROM wallets WHERE user_id = ${userId}::uuid FOR UPDATE
`;
const balanceBefore = wallet[0].balance;

// Step 5: Update
await tx.$executeRaw`UPDATE wallets SET balance = balance + ${amount} ...`;

const balanceAfter = balanceBefore + amountBigInt;

// Step 6: Create record dengan snapshot
await tx.walletTransaction.create({
  data: {
    ...
    balanceBefore,  // ← nilai dari step 4
    balanceAfter,   // ← dihitung: balanceBefore + amount
  }
});
